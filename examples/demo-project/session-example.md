# 示例会话

下面是一个推荐的首轮用户输入示例。

## 用户输入

请作为投标经理 Agent 接管当前工作区。先检查 `inbox/`、`output/`、`PROGRESS.json` 是否已经就绪，再确认当前项目输入和长期知识资产是否分开。若 `output/normalized/` 已有标准化产物，优先使用；若已有 `02-TENDER-PARSE.generated.md`，先审阅它而不是从零重复清点。然后完成项目启动咨询，再判断招标文件是否多包，生成解析页、证据缺口清单、评分点-章节-证据映射表和目录占位。证据不足时先报缺，不要直接进入完整正文起草。

## 预期 manager 行为

1. 明确当前阶段和已存在产物
2. 先问最少关键问题
3. 检查工作区是否存在 `inbox/`、`output/`、`logs/`
4. 检查是否已有标准化产物和 generated parse skeleton
5. 若工作区缺材料，只输出缺口和下一步动作
6. 进入招标解析阶段，判断是否多包
7. 若多包，先要求确认目标包件
8. 形成解析页与映射页
9. 形成证据缺口清单
10. 生成目录占位
11. 请求用户确认目录
12. 复杂项目中，将写作、合规核对、格式整理、质量审计分配给不同内部角色

## 预期质量标准

- 不在目录确认前写完整正文
- 优先读取标准化后的当前项目输入，而不是直接依赖二进制文件
- 已有 generated parse skeleton 时，先把它当作解析起点
- 证据不足时先报缺，不把缺口伪装成可交付内容
- 不把原厂能力写成投标人自有能力
- 不在无证据情况下输出正式资格结论
- 不把当前项目招标文件默认当成长期知识资产
- 不让同一个内部角色同时承担主要写作和最终审批
- 不暴露内部多 agent 编排细节给用户，除非用户主动询问
