# 示例会话

下面是一个推荐的首轮用户输入示例。

## 用户输入

请作为投标经理 Agent 接管当前工作区，优先处理当前项目输入目录。先检查当前项目输入是否已经标准化到 project-runs 下的 normalized 目录，以及是否已有 02-TENDER-PARSE.generated.md；若没有，请先按标准化清单记录风险。然后完成项目启动咨询，再判断招标文件是否多包，生成解析页、证据缺口清单、评分点-章节-证据映射表和目录占位。对于中大型项目，请在内部采用多 Agent 分工：写作、证据、合规核对、格式整理、质量审计相互分离，最终由 bid manager 汇总把关。

## 预期 manager 行为

1. 明确当前阶段：项目启动
2. 先问最少关键问题：
   - 投标主体是谁
   - 本次角色是什么
   - 必须纳入哪些厂商/证据
   - 哪些不能写太满
3. 检查工作区是否存在 `bid-vault/`
4. 检查当前项目输入文件夹是否存在
5. 检查是否已有标准化产物，若无则明确记录
6. 检查是否已有 generated parse skeleton，若有则先审阅
7. 进入招标解析阶段，判断是否多包
8. 若多包，先要求确认目标包件
9. 形成解析页与映射页
10. 形成证据缺口清单
11. 生成目录占位
12. 请求用户确认目录
13. 复杂项目中，将写作、合规核对、格式整理、质量审计分配给不同内部角色

## 预期质量标准

- 不在目录确认前写完整正文
- 优先读取标准化后的当前项目输入，而不是直接依赖二进制文件
- 已有 generated parse skeleton 时，先把它当作解析起点而不是重复从零整理文件清单
- 证据不足时先报缺，不把缺口伪装成可交付内容
- 不把原厂能力写成投标人自有能力
- 不在无证据情况下输出正式资格结论
- 不把当前项目招标文件默认当成长期知识资产
- 不让同一个内部角色同时承担主要写作和最终审批
- 不暴露内部多 agent 编排细节给用户，除非用户主动询问
